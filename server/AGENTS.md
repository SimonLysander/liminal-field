# Repository Guidelines

## 项目结构与模块组织

本仓库是 `LIMINAL FIELD` 的 NestJS 服务端，定位为内容与 AI 能力之间的 BFF / 业务后端，而不是单纯博客 API。当前代码集中在 `src/`，包含 `main.ts` 启动入口和基础 `app.*` 模块。后续新增能力时，按领域拆分目录，如 `src/content/`、`src/navigation/`、`src/files/`、`src/ai/`。单元测试统一放在模块下的 `__tests__` 目录中，使用 `*.test.ts` 命名；端到端测试保留在未来的 `test/` 目录。

## 构建、测试与开发命令

统一使用 `pnpm`：

- `pnpm install`：安装依赖。
- `pnpm start:dev`：本地开发，启用 watch。
- `pnpm build`：编译到 `dist/`。
- `pnpm start:prod`：运行构建结果。
- `pnpm test`：执行 Jest 单测。
- `pnpm test:cov`：生成覆盖率报告。
- `pnpm lint`：执行 ESLint 并修复可自动修复的问题。
- `pnpm format`：格式化 `src/**/*.ts` 与 `test/**/*.ts`。

## 代码风格与命名约定

使用 TypeScript 和 NestJS 标准命名：`*.module.ts`、`*.controller.ts`、`*.service.ts`。Prettier 采用单引号和尾随逗号，保持现有 2 空格缩进。类名使用 PascalCase，变量和方法使用 camelCase，DTO、实体、枚举按语义命名，避免 `data`、`item` 这类泛化名称。对外接口优先围绕领域建模，例如 `ContentItem`、`StructureNode`，不要直接暴露底层存储结构。

## 测试规范

测试框架为 Jest，`*.test.ts` 统一放在模块下的 `__tests__` 目录中。优先测试控制器和服务的公开行为，避免把测试绑死在内部实现。新增模块至少补充服务层单测；涉及路由、导入流程或异步任务时，补充 e2e 或集成测试。提交前至少运行 `pnpm test` 和 `pnpm lint`。

## 架构约束

服务端需要支撑“主题树导航 + 统一内容模型 + 文件导入 + AI 扩展”四条主线。导航结构与内容实体必须解耦：目录树负责组织关系，内容服务负责文档、图文和后续内容类型的抽象。设计 API 时优先考虑前端首页聚合、文档阅读、画廊流和 AI 摘要场景，不要只做底层 CRUD。

## 提交与 Pull Request 规范

延续现有 Conventional Commits 风格，例如 `feat: add content module`、`fix: handle missing content path`。PR 需要写清变更目的、核心实现、测试结果和潜在影响；若改动接口、页面聚合结构或配置项，请附示例请求 / 响应或迁移说明。

## 配置与安全提示

服务端口来自 `PORT`，默认 `3000`。不要提交 `.env`、密钥或真实凭证。新增外部依赖时，在 PR 中注明用途、运行要求和本地开发替代方案；涉及对象存储、数据库、向量库时，优先通过环境变量配置。

# Agent 开发准则

1. 至少先进行的足够的思考再行动，需要进行有深度、有广度的思考
2. 注释与代码同行：当你写下某段代码，你也应该简述说明【如果需要】为了解决什么问题、【如果需要】要做什么、【如果需要】作了什么样的设计，当你要更新代码时也不应该忘记更新注释
3. 时刻以务实犀利的态度讨论方案、设计、需求，不能一味迎合我，一切为了高质量、高扩展的优秀项目；涉及一些关键的、影响项目质量的方案、设计、实现，你应该主动询问我以消除未知、不确定的因素，避免因为其导致项目推进和质量出现问题
4. 你可以把反复踩的、关键的、影响项目推进的坑持续迭代在 Agents.md
5. 当方案确定之后，遵循优秀的开发模式：设计模式、开发、单元测试、lint校验等等

### Agent 写代码注释要求

- Agent 在本仓库写代码时，关键实现必须同时写解释性注释。
- 注释至少应覆盖以下信息中的一项或多项：
- 这段代码要解决什么问题
- 为什么采用当前策略
- 这个实现约束了什么边界
- 这段实现规避了什么坑或兼容了什么限制
- 简单、语义自明的代码不强制加注释，但关键流程、复杂判断、协议处理、边界校验、设计取舍必须写。
- 禁止只写翻译代码表面的废话注释，例如“给变量赋值”“调用方法获取结果”。
- 当 Agent 修改已有关键实现时，如果原注释已经不准确，必须同步更新，而不是保留过期说明。

### Agent 动手前说明要求

- Agent 在进行非小改动前，应先简要说明要解决什么问题、准备怎么拆、主要风险点是什么，再开始实施。
- 如果改动会影响接口、数据模型、模块边界或关键流程，不能跳过这一步。

### Agent 不确定时先确认

- Agent 遇到会影响项目质量的关键不确定项时，必须先提出问题，不允许直接脑补后落地。
- 尤其是涉及数据模型、接口语义、模块职责、持久化策略、协议规则时，应优先确认。

### Agent 改动与验证要求

- Agent 修改接口、返回结构、关键业务行为时，必须同步更新测试；必要时同步更新相关文档或注释。
- Agent 默认不能以“先写功能，后补测试”的方式结束任务。
- Agent 在认为任务完成前，应主动完成基本验证，如 test、lint、build；如果因环境或依赖问题无法完成，必须明确说明阻塞点。

### Agent 边界与类型要求

- Agent 不得为了省事混淆 controller、service、repository 职责。
- Agent 不得随意使用 `any`、`as any`、`as never` 这类方式掩盖设计或类型问题。
- 如果某段实现需要靠强制断言才能成立，应优先反思设计是否有问题，而不是先压过类型系统。
